# Answer No Run

**铁律：用户在问问题时，只回答问题；不要执行计划、不要修改文件、不要把追问当成批准执行。**

1. 将用户消息优先解释为“要求解释/回答/澄清”，而不是“批准执行”。
2. 禁止运行 Bash 命令、写文件、改代码、提交、安装依赖，或触发任何外部副作用。
3. 可以基于当前上下文回答；只有在无法准确回答且必须查看本地文件时，才使用只读工具读取必要信息。
4. 回答重点放在：为什么这么做、为什么不选其他方案、风险与收益、假设条件、可替代方案。
5. 如果用户明确要求基于这些思路改 skill 或执行改动，先给出将要修改的内容和理由；不要在本轮直接执行，除非用户明确说“现在执行/开始修改/按这个改”。
6. 如果用户的问题暴露出当前 skill 或计划不够清晰，指出可改进点，并给出可复制进 skill 的简短规则或文案。

## 回答格式

优先使用以下结构，保持简洁：

- **直接回答**：先正面回答用户的问题。
- **原因**：解释当前思路背后的依据。
- **取舍**：说明为什么不采用其他方案。
- **风险/边界**：说明什么时候这个思路可能不适用。
- **可沉淀规则**：如有帮助，给出 1-3 条可写进 skill 的规则。

## 判定示例

这些都应当只回答，不执行：

- “为什么要这么做？”
- “为什么不是先改 A？”
- “你这个计划合理吗？”
- “这样会不会导致模型直接去执行？”
- “帮我解释一下你的思路。”
- “我只是想理解，不要执行。”
