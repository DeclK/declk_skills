# Note Organize — 笔记串联整理规范

## 背景

在学习/研究过程中，用户会产出大量分散的笔记点——每个点解决一个具体问题。这些点在探索过程中自然形成：有些浅尝辄止，有些被跳过，有些沿着某条线索深挖下去。当学习告一段落时，这些点仍然处于离散状态，需要串联成一篇结构清晰、叙述流畅的文档。

这个串联过程本质上是机械性的（归并、排序、加过渡），完全可以由 agent 自动化完成。

## 核心约束

1. **不修改已有内容** — 只做排版和叙述层面的重组，原文一字不动
2. **添加总结性开头** — 在笔记最前面用 2-4 句概括全文脉络
3. **允许少量过渡语句** — 在章节或话题之间加入承上启下的简短连接

## 模式判定

在开始整理前，先判定当前属于哪种模式：

### 自动归并模式（默认）

用户只提供了笔记内容，没有给出大纲。此时按 Step 1-6 的完整流程执行，从笔记中自动发现主题结构。

触发条件：用户消息中没有出现大纲/提纲/目录/框架等结构化的章节列表。

### 大纲引导模式

用户同时提供了**大纲**和**笔记**。大纲通常是一个文件路径（如 `outline.md`）或用户消息中直接给出的结构化章节列表。

大纲是一个预设的章节框架，定义了最终文档应包含哪些章节及其顺序。笔记则是需要填入这个框架的原始内容。

触发条件（满足任一）：
- 用户显式指出某个文件是大纲：「按照 outline.md 的结构整理 notes.md」
- 用户消息中包含一个明显的结构化章节列表（如编号的标题列表），并说明了「按此大纲整理」
- 用户在 `/note-organize` 之后提供了两个文件引用，并说明它们的关系

**当识别到用户提供了大纲时，使用「大纲引导模式」的工作流程（见下方），而非默认的自动归并流程。**

## 共享步骤（两种模式共用）

以下步骤在两种模式中都会用到，只是前置的「结构来源」不同——自动归并模式通过 Step 1-3 自建结构，大纲引导模式通过 Step B1-B4 从大纲获取结构。

### 共享 Step S1: 添加总结性开头

在笔记最前面写一个简短的总结段落（2-4 句），包含：
- 本文研究的主题/要解决的问题
- 覆盖的主要方面
- （可选）读者看完后能获得什么理解

大纲引导模式下额外注意：
- 如果大纲与实际笔记有偏差，在开头简要说明
- 如果有补充章节，在开头提及

### 共享 Step S2: 添加过渡语句

在以下位置考虑添加过渡句：
- 章节与章节之间：承上启下
- 话题发生显著跳跃处：从 A 视角切换到 B 视角
- 从动机/背景进入具体方案处
- 从方案描述进入实现细节处

过渡句要求：
- **简短** — 1-2 句
- **自然** — 承接上文核心信息，引出下文关注点
- **克制** — 不需要每两个点之间都加。如果两个点主题紧密、自然衔接，就不加

大纲引导模式下，章节之间的过渡句应反映大纲本身设计好的逻辑关系。

### 共享 Step S3: 格式化输出

将整理后的内容写入**新文件**，原文件不做任何修改。

**输出格式取决于原始笔记的形态。**当原始笔记是离散的短笔记点（每条一两句话、无内部结构）时，可用 `>` 引用标记每条笔记：

```
<总结性开头 — 新添加>

## <章节标题 1>

> <原始笔记点 A — 原文不变>

<过渡句 — 仅当需要时添加>

> <原始笔记点 B — 原文不变>

## <章节标题 2>

...
```

当原始笔记本身已经是包含代码块、表格、子标题、加粗段落等丰富排版的完整文档时，**绝对不要**用 `>` 将其包裹——这会被 Markdown 渲染为引用块，破坏代码块、表格、嵌套列表等原有格式。此时应保持原始排版原样搬入章节，仅在其外围添加章节标题和过渡句，不加任何额外标记。两种情况的本质区别：前者是对「笔记点列表」做整理，后者是对「文档段落」做重组。判断标准——如果原文里已经有三个以上的反引号代码块或 pipe 表格，就走段落模式，不加 `>`。

命名规则：
- 如果笔记来自文件：`<笔记文件名>_organized.md`（例如 `notes.md` → `notes_organized.md`）
- 如果笔记是用户消息中的内联文本：命名需向用户确认，默认为 `notes_organized.md`

## 工作流程（自动归并模式）

当用户**仅提供笔记内容**时，使用此模式。核心任务是从笔记中自动发现主题结构。

### Step 1: 通读与拆解

- 通读用户提供的全部笔记内容
- 识别每个独立的笔记点（一个点 = 一个具体问题及其解答/分析）
- 提炼每个点的核心命题（一句话概括这个点在说什么）

### Step 2: 归并与分组

将独立的笔记点按以下维度归并：

- **主题相近** → 讨论同一主题/同一模块的点归为一组
- **逻辑递进** → 有因果或依赖关系的按先后排列
- **从浅到深** → 基础概念在前，深入分析在后
- **从泛到专** → 通用背景在前，具体问题在后

每个分组形成一个候选章节，给每个章节一个概括性的标题。

### Step 3: 确定全局叙事结构

在归并后的分组之间建立全局顺序，常见模式：

- **背景 → 问题 → 方案 → 细节 → 总结**
- **概览 → 模块 A → 模块 B → 模块间交互 → 端到端流程**
- **动机 → 核心思想 → 关键实现 → 边界情况 → 未解决问题**

选出最适合这批笔记内容的叙事顺序。

### Step 4-6: 执行共享步骤

依次执行 [共享 Step S1](#共享-step-s1-添加总结性开头)、[S2](#共享-step-s2-添加过渡语句)、[S3](#共享-step-s3-格式化输出)，完成整理和输出。

## 工作流程（大纲引导模式）

当用户同时提供了**大纲 + 笔记**时，使用此模式。大纲是最终文档结构的权威来源，核心任务不是「发现结构」，而是「将笔记映射到大纲上」。

### Step B1: 解析大纲

- 如果大纲来自文件，通读大纲文件，理解其章节结构和层级关系
- 如果大纲是用户消息中的内联文本，提取其章节列表和层级
- 识别每个大纲章节的**意图**：这个章节想覆盖什么主题/回答什么问题
- 只要大纲本身没有明显的逻辑矛盾或严重缺失，**以大纲为准**不动摇。如果大纲有明显问题（如两个章节完全重复），先向用户确认

### Step B2: 通读笔记并拆解

- 通读全部笔记内容
- 识别每个独立的笔记点
- 提炼每个笔记点的核心命题（一句话概括）
- **额外步骤**：为每个笔记点标注关键字/主题标签，便于后续映射

### Step B3: 将笔记映射到大纲章节

将每个笔记点分配到最合适的大纲章节：

- **按主题匹配** → 笔记点的核心命题与某个大纲章节的意图一致
- **按关键词匹配** → 笔记中的术语、模块名与章节标题或描述对应
- **按逻辑位置匹配** → 笔记讨论的内容在逻辑上属于某章节的覆盖范围

映射时遵循以下规则：
- 一个笔记点只放入一个章节（即使它可能和多个章节相关，选最匹配的那个）
- 同一章节内的笔记点按逻辑递进或从浅到深的顺序排列
- 大纲章节的标题作为最终文档的 `##` 标题直接使用（不加编号）

### Step B4: 处理映射中的边缘情况

按优先级处理：

**(1) 无笔记对应的章节**
- 如果某个大纲章节完全没有笔记点可以映射，在最终文档中保留该章节标题，并在标题下添加一行简短说明：`> *（此处暂无对应笔记）*`
- 不要在章节内编造内容

**(2) 无法归入任何章节的笔记点（孤立笔记）**
- 如果笔记点无法匹配任何大纲章节，将它们收集到一个独立的 `## 补充笔记` 章节（放在文档末尾）
- 如果孤立笔记数量较多（≥5个），尝试对它们进行归并，生成一个或多个额外的补充章节
- **在总结性开头中提及存在补充章节**，说明这些内容超出大纲覆盖范围但在笔记中出现

**(3) 大纲与实际笔记内容存在偏差**
- 如果大纲的章节顺序与笔记内容的逻辑依赖关系明显矛盾，可以向用户提出调整建议，但**最终以用户确认为准**
- 如果大纲覆盖的主题与笔记实际讨论的主题不完全一致，在整理时以笔记实际内容为准进行映射，同时在总结性开头中说明本文的实际覆盖范围可能与大纲有出入

### Step B5-B7: 执行共享步骤

依次执行 [共享 Step S1](#共享-step-s1-添加总结性开头)、[S2](#共享-step-s2-添加过渡语句)、[S3](#共享-step-s3-格式化输出)，完成整理和输出。

## 规则

### 规则 1: 原文绝对不动

**绝不修改**用户已有的笔记内容。包括：
- 不改写句子或调整措辞
- 不增删技术细节
- 不修正拼写或术语（除非用户明确要求）
- 不改变代码块内容
- 不改变原文中的格式（编号、缩进、强调标记等）

### 规则 2: 总结性开头要精炼

- 只写一个段落，2-4 句
- 说清「研究什么、涵盖什么」，不需要展开背景介绍
- 风格与用户原文保持一致——如果原文简洁，开头也简洁；如果原文细致，开头可稍长

### 规则 3: 过渡语句要克制

- 只在大话题切换处添加
- 不要为加过渡而加过渡——自然衔接优于强行过渡
- 用中文撰写，与用户笔记语言一致

### 规则 4: 保留原始结构线索

- 用户原文中的编号、标签、时间戳等，完整保留
- 用户原文中的 `/ai` 标记、`TODO`、`FIXME` 等，保持原样不动

### 规则 5: 不确定时询问用户

遇到以下情况先确认：
- 两个笔记点的逻辑关系有多种合理理解
- 某个笔记点同时适合多个分组（自动归并模式）或同时适合多个大纲章节（大纲引导模式）
- 笔记间存在明显矛盾（可能是用户不同阶段的理解变化）
- 某个笔记点过于孤立，找不到合适的归属
- **（大纲引导模式）大纲章节与笔记内容严重不匹配**，导致大量笔记无法映射或大量章节无内容
- **（大纲引导模式）大纲本身存在结构性问题**（如章节重复、严重遗漏），影响整理质量

### 规则 6: 章节标题不加序号

新增的顶层章节标题（`##` 级别）**不要**使用中文数字（一、二、三）或阿拉伯数字进行编号。标题本身的内容就足以表达章节边界，编号冗余且容易在后续增删章节时产生不一致。

### 规则 7: 大纲优先（仅大纲引导模式）

在大纲引导模式下：
- **大纲的章节结构和顺序是权威的**，不自行增删或重排大纲中的章节
- 大纲章节标题原样使用（但去掉原大纲可能已有的编号，遵循规则 6）
- 笔记内容服务于大纲结构，而非反过来——不要因为笔记内容分布不均而擅自调整大纲
- 唯一的例外：孤立笔记可以添加补充章节（见 Step B4-(2)）

### 规则 8: 空章节如实标注（仅大纲引导模式）

- 如果某个大纲章节没有任何对应的笔记内容，保留标题并标注 `> *（此处暂无对应笔记）*`
- 不编造内容、不从其他章节挪内容来填充空章节
- 空章节的存在本身就是信息——它告知用户哪些大纲内容在笔记中没有覆盖到
